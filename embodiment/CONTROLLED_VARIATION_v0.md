# IAMMAI Controlled Variation v0

## 1. Purpose

This file defines the first controlled-variation boundary for IAMMAI after the first lawful local proof.

Its job is to state how the next embodiment phase may test repeatability across more than one bounded matter without widening the slice that has already been proved.

It does not yet try to define:

- a new matter category
- a new runtime family
- a richer witness ecology
- separately preserved threshold-state handling
- a broad new embodiment design
- a product plan
- a public rollout posture

It is a bounded embodiment-governance document for the phase immediately after first proof.

## 2. First Principle

The first proof remains historically fixed.

It remains exactly what it proved: one bounded local lawful cycle on one bounded matter, under the first embodiment posture, with its own historical artifacts.

Controlled variation does not rewrite that proof. It tests whether the same slice class can be repeated across multiple bounded matters without changing what the original proof was.

Variation must remain smaller than scope expansion.

If a proposed change requires a new matter class, a new runtime role family, a new canonical artifact family, broader access, or richer ontology, it is not controlled variation. It is a later phase and should be named as such.

Lawful embodiment therefore remains bounded even as variation begins.

## 3. Why Controlled Variation Is Needed

One successful local proof is enough to show that a lawful cycle can exist.

It is not yet enough to show that the same bounded cycle can repeat across more than one matter of the same class.

Repeatability across multiple bounded matters matters because the next question is narrower than expansion:

- whether the same lawful body can carry the same slice class more than once
- whether the first proof depended on one exact bounded note only
- whether the role family, artifact family, and failure posture remain stable when the matter changes but the slice does not

Variation must be governed rather than improvised because unguided variation is the fastest path to hidden widening. New matter types, new access scope, product logic, identity-heavy content, or richer ontology can otherwise enter under minor-change language.

This phase comes before richer ontology changes because the existing slice should be tested for repeatability before the repo begins changing the slice itself.

## 4. What May Vary

Controlled variation in this phase is conservative.

The following may vary:

- the matter identifier, provided each matter remains separately bounded
- the exact note content, provided the matter remains one bounded local note in its exact received form
- bounded note-level metadata, but only where it is justified for local differentiation or audit and does not change the matter class or runtime class

Where note-level metadata is used, it must remain:

- local
- bounded
- subordinate to the note itself
- non-commercial
- non-identity-heavy
- incapable of smuggling in broader product or domain semantics

Repeatability in this phase is established by running the same slice separately across multiple bounded matters.

It is not established by widening one run into multi-matter handling.

## 5. What Must Remain Invariant

The following must remain invariant across the controlled-variation phase:

- the matter class remains the bounded local note matter
- each run remains one governed truth-formation attempt for one threshold-facing bounded matter
- the runtime cycle class remains the same first slice class rather than a broadened execution path
- the runtime role family remains Interface Gateway, Validator, Witness Emitter, Governor, State Engine, and Registry / Store
- the canonical artifact family remains validation artifact, witness artifact, governance action record where produced, transition record where produced, state record where produced, and registry-preserved canonical records or visible canonical absence where they are not lawfully produced
- the authoritative versus derivative boundary remains intact
- the local execution boundary remains bounded to the embodiment workspace
- success and failure integrity remain the same, including visible non-passage, no role collapse, and no cosmetic substitution
- the execution posture remains local, internal-first, controlled, auditable, and non-public

In practical terms, controlled variation may change the bounded matter inside the slice. It may not change the slice into a different kind of embodiment.

Threshold-facing here defines the slice class only. Each submitted matter must still pass the required runtime validation during the run.

## 6. What Controlled Variation Is Not

Controlled variation is not:

- a new matter category
- historical artifact ingestion
- product data ingestion
- identity-heavy matter
- multi-party interaction
- public rollout
- broadened access scope
- a richer trace model disguised as simple variation
- a hidden expansion of runtime roles or canonical artifact families

If a proposed next move falls into one of these categories, it should not be carried under the label of controlled variation.

## 7. Why Threshold/Source and Witness-Richness Are Not the Next Fixes

The first proof contains two known seams.

First, the first slice used threshold as the referenced source relation for truth formation rather than preserving a separate threshold state artifact.

Second, the first slice used validation-adjacent witness only rather than a fuller multi-event witness family.

These seams remain known and valid.

They do not erase the first proof, and they do not justify retroactive rewriting of its artifacts.

They are not the next immediate change because controlled variation asks a narrower question first: can the same historically fixed slice repeat across multiple bounded matters without hidden widening.

If richer threshold/state handling or richer witness handling were introduced now, the repo would be changing the slice while trying to test repeatability of the slice. That would blur the historical integrity of the first proof.

Controlled variation therefore comes before richer state or witness expansion.

## 8. Success Criteria for the Variation Phase

The controlled-variation phase counts as successful when:

- multiple bounded matters can each run through the same lawful cycle class
- each run remains one matter, one bounded note, and one bounded cycle
- the canonical artifact family remains intact across successful and non-successful runs
- runtime role separation remains intact
- failure remains visible and typed rather than being cosmetically smoothed
- the first proof remains historically untouched and separately recoverable
- no hidden broadening of matter class, access scope, semantics, or embodiment posture occurs

Success in this phase is repeatable bounded execution, not broader embodiment.

## 9. Failure or Stop Conditions for the Variation Phase

The controlled-variation phase should pause, stop, or be judged as drifting if:

- repeatability requires a new matter category
- repeatability requires broader local, repo, machine, or network access
- repeatability requires product logic or domain-specific behavior
- repeatability requires identity-heavy matter
- repeatability requires multi-party interaction
- runtime roles begin to collapse or be bypassed
- canonical artifact expectations become unstable
- richer threshold/state handling or richer witness handling is being introduced implicitly rather than named as a new phase
- metadata begins carrying broadened semantics rather than bounded note-level differentiation
- multiple matters are being folded into one run rather than kept as separately bounded cycles

At that point the lawful move is not to smuggle the widening through. The lawful move is to stop, name the drift clearly, and decide the next phase explicitly.

## 10. Relation to Future Steps

Once controlled variation is stable, later questions may include:

- a richer witness family
- richer threshold or separately preserved threshold-state handling
- deeper continuity or SPIRAL questions

Those are later questions.

They should enter as explicit later steps, not as hidden content inside controlled variation.

## 11. Summary

This document defines the first post-proof embodiment boundary of IAMMAI as governed repeatability of the same bounded slice across multiple separately bounded local note matters.

Its purpose is to let the body test repeatability without retroactively widening the first proof, without smuggling in richer ontology, and without allowing variation to become hidden scope expansion.

**One-Line Definition:** `CONTROLLED_VARIATION_v0.md` defines the first lawful controlled-variation boundary of IAMMAI as repeatable execution of the same bounded local note slice across multiple separately bounded matters while keeping matter class, runtime roles, artifact family, access scope, and public posture invariant.
