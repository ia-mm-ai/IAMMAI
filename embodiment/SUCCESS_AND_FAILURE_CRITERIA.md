# IAMMAI Success and Failure Criteria

## 1. Purpose

This file defines the bounded criteria by which the first lawful local cycle will be judged as:

- successful
- partially completed
- failed
- non-passing
- non-proceeding

Its job is to make explicit what must exist, what must remain visible, and what must not be cosmetically substituted in order for the first embodiment to count.

It does not yet try to define:

- code
- incident response
- product-style QA workflow
- operational escalation policy
- infrastructure monitoring design
- generalized judgment rules for later embodiments

It is a bounded embodiment-judgment document.

## 2. First Principle

Success must be judged by lawful integrity, not presentation quality.

Visible failure is preferable to false continuity.  
The first cycle counts only if the required distinctions survive in motion.  
Success and failure criteria must serve the constitutional core.

In execution terms, that means the first proof is valid only when the cycle can be judged from lawful runtime behavior and canonical artifacts rather than from interface appearance, operator confidence, or narrative summary.

## 3. Why Explicit Criteria Are Needed

A first live cycle needs explicit judgment criteria.

- Without explicit criteria, the first run can be declared successful merely because it looked plausible.
- `It seemed to work` is not enough because lawful embodiment depends on typed roles, typed artifacts, and visible non-passage where the cycle does not complete.
- Failure visibility must be part of the definition because a first embodiment that hides failure is not a lawful proof of the body.
- A lawful cycle must be auditable after execution so later readers can see what occurred, what did not occur, and what canonical traces actually exist.

The criteria in this file therefore constrain how the first cycle is to be judged, not merely how it is hoped to appear.

## 4. Success Criteria

The first cycle counts as successful only if all of the following are true:

- bounded input or contact occurred on the selected matter
- required validation ran
- witness emitted where constitutionally required in the selected cycle
- governance acted where required
- state and transition outputs were produced where applicable
- the Registry / Store preserved the canonical records actually required by the cycle
- the runtime path remained reconstructable afterward
- no role collapse occurred
- no silent substitution occurred

For the selected first cycle, that means success requires:

- a validation artifact showing the required checks passed
- the required witness artifact or artifacts
- a governance action record authorizing the move
- a transition record for the lawful move into truth
- a state record for the resulting truth state
- registry-preserved canonical records for the artifacts actually produced

If any required artifact is absent, silently substituted, or only derivative, the run does not count as full success.

## 5. Partial-Completion Criteria

The first cycle counts as partially completed when some bounded runtime stages occurred lawfully, but the cycle cannot yet count as a completed lawful cycle.

Partial completion includes cases such as:

- bounded input occurred, validation ran, and witness emitted, but governance did not proceed where governance is required
- validation, witness, governance, and state handling occurred, but canonical preservation failed or remained incomplete
- the runtime path exists in part, but the required final artifact set for a successful cycle does not exist

Partial completion is not full success.

A partially completed cycle may still prove that some runtime roles operated lawfully, but it does not prove that the selected first cycle completed as a fully preserved lawful embodiment.

## 6. Failure and Non-Passage Criteria

The first cycle counts as failed, non-passing, or non-proceeding when one or more of the following occur:

- validation non-passage
- missing or incomplete witness where witness is constitutionally relevant
- governance non-progression where governance is required
- failed or blocked state movement
- failed canonical preservation
- incomplete runtime sequence that is cosmetically represented as complete

The judgment distinctions are:

- `non-passing`
  The cycle did not proceed because validation did not pass in typed form.
- `non-proceeding`
  The cycle did not proceed because governance denied, governance held, or another required runtime condition did not lawfully permit continuation.
- `failed`
  The cycle should have produced or preserved something required by the selected success path, but the runtime did not complete that requirement.

These states must remain distinct where the cycle actually distinguishes them.

## 7. What Must Remain Visible

On any non-success path, the following must remain visible:

- where the cycle stopped
- which role did not proceed
- which artifact is missing
- whether the outcome was fail, hold, deny, blocked, or indeterminate where relevant
- whether canonical preservation did not occur

Invisible failure invalidates the proof value of the slice.

If the run reaches a non-success outcome but does not preserve that outcome visibly enough to reconstruct the stopping point and missing artifact relation, the embodiment cannot be counted as a valid proof of lawful motion.

## 8. Artifact-Based Judgment

Success and failure are judged through artifacts.

The artifact roles in this judgment are:

- `validation artifact`
  Shows whether the required checks ran, what they targeted, and what outcome they produced.
- `witness artifact`
  Shows that constitutionally relevant contact and runtime occurrence were actually preserved as witness rather than merely remembered or logged.
- `governance action record`
  Shows whether lawful force authorized, denied, or held the selected move.
- `transition record`
  Shows whether lawful movement occurred and what move it was.
- `state record`
  Shows what standing state resulted where the selected success path lawfully formed truth.
- `registry-preserved canonical records`
  Show whether the produced artifacts became durably preserved rather than merely observed or displayed.

Success cannot be asserted without the required artifact set for the chosen slice.

Likewise, non-success must still be judged from the artifacts that do exist, together with the visible absence or incompleteness of those that do not.

## 9. Anti-Collapse Judgment Rules

The following must never be counted as success:

- UI or display output without canonical preservation
- validation pass without required downstream movement
- governance appearance without governance record
- witness-like trace without bounded witness artifact
- state mutation without typed transition and state records
- registry absence hidden by derivative display
- narrative claim of completion without reconstructable runtime path

If the cycle relies on any of these substitutions, it does not count as a lawful first proof.

## 10. What This First Proof Does and Does Not Require

The first proof does require lawful runtime integrity.

It does not require:

- broad deployment
- polished UI
- scale
- public release

It does require enough runtime reality to count as embodiment rather than theory.

That means the first proof must show real bounded input, real validation, real witness, real governance where required, real state handling where applicable, and real canonical preservation or visible preservation failure.

## 11. Open Questions

The following remain intentionally open before these criteria are used in an actual run:

- the exact local observation surface
- the exact artifact inspection method
- the exact operator workflow
- the exact host implementation details

These are embodiment-build questions, not constitutional ambiguities.

This document defines the criteria by which the first lawful local cycle of IAMMAI is judged as successful, partially completed, failed, non-passing, or non-proceeding on the basis of lawful runtime integrity and canonical artifacts.

Its purpose is to ensure that the first embodiment is judged by reconstructable motion, visible non-passage, and preserved distinctions rather than by presentation quality or narrative confidence.
