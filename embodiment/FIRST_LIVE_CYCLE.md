# IAMMAI First Live Cycle

## 1. Purpose

This file defines the first actual lawful local cycle to be executed for IAMMAI.

Its job is to turn the embodiment decision and the chosen matter into one concrete, bounded, auditable runtime cycle.

It does not yet try to define:

- code
- a deployment plan
- a UI spec
- a product flow
- broad environment behavior
- generalized runtime execution

It is a bounded embodiment execution-definition document.

## 2. First Principle

The first live cycle must be real enough to count.

It must remain bounded.  
It must preserve typed runtime distinctions.  
It must remain auditable.  
It must keep failure visible rather than cosmetically smoothing it.

In execution terms, that means the cycle must operate through the real minimum runtime body in one local bounded attempt rather than through narrative substitution, product surface illusion, or partial role collapse.

## 4. Chosen Cycle

The exact first lawful local cycle is:

**one internal-first governed truth-formation attempt for one bounded local note matter already placed in threshold-eligible condition, concerning whether that note may lawfully form truth in its exact received form.**

This cycle defines:

- the selected matter: one bounded local note matter with one matter id and one submitted note
- the initiating input or contact: one bounded local submission of that note into the selected cycle
- the runtime roles that must participate: Interface Gateway, Validator, Witness Emitter, Governor, State Engine, and Registry / Store
- the canonical artifacts that must result if the cycle succeeds: validation artifact, witness artifact, governance action record, transition record, state record, and registry-preserved canonical records
- the visible non-completion states if the cycle does not succeed: validation non-passage, missing or incomplete witness, governance denial or HOLD, failed or blocked state movement, and failed or incomplete canonical preservation

This is one cycle only. It is not a general-purpose local platform and not a broad runtime environment.

## 5. Runtime Participants in This Cycle

The participating runtime parts are:

- Interface Gateway
  A minimal local operator surface that receives the bounded note input and routes it into the cycle.
- Validator
  Performs the required threshold, transition-legality, and governance-precondition checks for the selected move.
- Witness Emitter
  Emits the witness trace required for constitutionally relevant contact and checking, and any later governance or transition witness trace that is actually emitted.
- Governor
  Applies authorize, deny, or HOLD to the truth-formation attempt.
- State Engine
  Performs lawful truth formation if authorized and not held, or preserves visible non-movement if not.
- Registry / Store
  Preserves the canonical records actually produced by the cycle.

All required parts are real, even if minimal.

## 6. Entry Conditions

Before the cycle begins, the following must already be true:

- the matter is defined in bounded form
- the submitted note exists in exact received form for this cycle
- the matter is selected for threshold-facing handling in this first live cycle
- the runtime components required by the selected slice are available
- the artifact expectations and schema family are known
- the cycle is being run in the chosen internal-first controlled context
- canonical preservation is available as a real registry surface for the records actually produced

This scope choice does not itself establish threshold satisfaction for the exact submitted note. The exact submitted note must still pass the required validation before lawful progression continues.

If these conditions are not present, the cycle must not be cosmetically treated as started.

## 7. Lawful Sequence

The cycle proceeds in the following high-level order:

1. A bounded local note input enters through the minimal local Interface Gateway in relation to one selected matter id.
2. The Interface Gateway binds that input to the selected matter and routes the truth-formation attempt into bounded protocol handling.
3. The Validator performs the required threshold, transition-legality, and governance-precondition checks for that exact note matter.
4. The Validator produces a validation artifact with typed outcome and relation to the selected target and condition set.
5. The Witness Emitter emits trace for the input-contact and the validation occurrence where constitutionally relevant.
6. The Governor receives the bounded request together with the required validation basis and resolves the move by authorizing, denying, or applying HOLD.
7. If the Governor authorizes and does not hold the move, the State Engine performs the threshold-eligible-to-truth transition and produces the corresponding transition and state outputs.
8. If validation does not pass, governance denies, governance holds, or the move is otherwise blocked, the cycle preserves typed non-movement rather than rewriting it as continuity.
9. The Witness Emitter may also emit governance-related or transition-related witness trace where such trace is actually emitted.
10. The Registry / Store preserves the canonical records actually produced by the cycle and makes any preservation failure visible if canonical durability does not complete.

The cycle ends only when the resulting canonical record set or visible non-completion state is reconstructable.

## 8. Canonical Artifacts Expected on Success

After a successful cycle, the minimum canonical artifacts expected are:

- one validation artifact showing the required checks passed
- at least one witness artifact preserving the constitutionally relevant contact and checking trace, plus any additional witness artifacts actually emitted for governance or transition occurrence
- one governance action record showing authorization of the truth-formation attempt
- one transition record preserving the lawful move into truth
- one state record preserving the resulting truth state
- registry-preserved canonical records for all artifacts actually produced in the cycle

No additional artifact family is required for this first live cycle beyond what the selected cycle actually uses.

## 9. Success Conditions

For the cycle to count as a successful lawful execution, the following must be true:

- the cycle ran as one bounded local attempt on the selected matter
- all required runtime roles participated in their own distinct roles
- the required validation passed in typed form
- required witness trace was emitted and preserved
- governance authorized the move and did not apply HOLD
- the State Engine produced the lawful truth-formation transition and resulting truth state
- canonical preservation occurred for the artifacts actually produced
- no silent substitution, summary-only replacement, or derivative display stood in for a required runtime role
- the completed cycle is reconstructable from canonical records

## 10. Failure Visibility Conditions

If the cycle does not complete or completes only partially, the following must remain visible:

- validation non-passage
- witness absence, incompleteness, or non-preservation
- governance non-progression where governance is required, including denial and HOLD
- failed or blocked state movement
- failed, incomplete, or missing canonical preservation

Visible failure is part of lawful embodiment, not its negation.

If the success path is not achieved but the cycle still preserves typed non-passage honestly and canonically where applicable, the run still proves something valid about the body. What invalidates the cycle is hidden failure, not visible non-completion.

## 11. What This Cycle Proves

A successful execution of this cycle would prove that:

- IAMMAI can execute one real local lawful cycle rather than remain only descriptive
- the minimum runtime body can operate as one bounded body on one bounded matter
- threshold does not silently collapse into truth
- validation, witness, governance, state handling, and registry preservation can coordinate without role confusion
- canonical artifacts can preserve the result of one live run
- the body can remain auditable after motion

## 12. What This Cycle Does Not Yet Prove

This cycle does not yet prove:

- the full preparation chain
- general multi-matter runtime behavior
- FINALIZE, INVALIDATE, or EVOLVE handling
- full continuity, decision, or consequence behavior
- broad deployment readiness
- production scalability
- public-facing operation
- full system completion

## 13. Internal-First Posture

This cycle remains:

- local
- controlled
- auditable
- non-market-facing
- non-privileged
- not yet a public rollout

It remains local so the first embodiment can stay bounded. It remains controlled so the proof is architectural rather than performative. It remains auditable so canonical records, non-passage, and failure visibility can be checked directly. It remains non-privileged because no private host, product, or operator acquires constitutional priority by running it first.
