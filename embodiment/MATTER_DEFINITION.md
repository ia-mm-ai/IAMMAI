# IAMMAI Matter Definition

## 1. Purpose

This file defines the first bounded matter for the first lawful local embodiment slice.

Its job is to state what the first matter is, why it was chosen, what makes it bounded enough, what makes it meaningful enough, what is included, what is excluded, and how it avoids product capture, private privilege, and unnecessary complexity.

It does not yet try to define:

- code
- a product spec
- a user story
- a market-facing scenario
- broad domain coverage
- a generalized matter library

It is a bounded embodiment-definition document.

## 2. First Principle

The first matter must be small enough to remain bounded.

The first matter must be real enough to count.  
The first matter must allow the full minimum runtime body to operate without distortion.  
The first matter must not grant private implementation privilege.

In execution terms, that means the chosen matter must be simple enough for one controlled local cycle, but still concrete enough that validation, witness, governance, state movement, and registry preservation are doing real work.

## 3. Why the First Matter Must Be Explicitly Defined

One bounded matter cannot remain vague.

- If the matter remains vague, the first embodiment can drift into arbitrary input handling rather than a selected lawful cycle.
- If the matter remains generic, the proof can become cosmetic because nothing specific was ever actually tested.
- If the matter is left implicit, the first embodiment can widen into public, commercial, or identity-heavy terrain without admitting that it has done so.

The first embodiment therefore needs a chosen matter rather than generic placeholder input. Matter definition protects against drift, overreach, and fake proof by fixing what the first cycle is actually about.

## 4. Chosen Matter

The chosen first matter is:

**one bounded local note matter concerning whether one single submitted note exists in the declared local form for that matter and may lawfully form truth in that exact received form.**

In execution-facing terms, this means:

- there is one matter id only
- there is one submitted note only
- the note is one bounded local input
- the note is non-commercial
- the note is non-identity-heavy
- the note is scoped only to the first embodiment run
- the truth-formation attempt concerns the note in its exact received form, not broad interpretation beyond that bounded form

This matter is suitable for internal-first controlled execution because it is neutral, auditable, and low-noise. It allows real contact, real checking, real witness trace, real governance where required by the selected slice, real state or transition effect where permitted, and real registry preservation.

It is chosen as appropriate for a threshold-facing first embodiment slice, but that choice does not itself establish threshold satisfaction in runtime. The exact submitted note must still pass the required validation during the actual run.

## 5. Why This Matter Was Chosen

This matter was chosen because it is small enough without being empty.

- It is small enough because it concerns one note, one matter id, one attempted truth-formation cycle, and one local bounded context.
- It still exercises the full runtime chain because the matter can be checked, witnessed, governed, transitioned, and preserved without relying on broad social participation or external market context.
- It avoids premature complexity because it does not depend on multi-party coordination, commercial activity, public identity claims, or domain-specific product behavior.
- It is better than broader or more public-facing first matters because it reduces legal, social, and operational noise while still forcing the architecture to prove itself in motion.

The matter is therefore intentionally plain. Its value is not in dramatic content. Its value is in being a lawful, auditable, non-trivial first matter for the body.

## 6. Included Boundaries

This matter definition includes:

- one bounded input
- one bounded local note
- one bounded matter id
- one bounded runtime cycle
- one governed truth-formation attempt for that note matter
- one canonical record family output set for what actually occurs

It also includes the condition that the note is handled only as the selected matter of the first local embodiment and not as a proxy for a wider product, social, or commercial domain.

## 7. Excluded Boundaries

This matter definition excludes:

- multi-party coordination
- broad public participation
- commercial flows
- identity-heavy flows
- domain-specific product behavior
- UI-first demonstration
- scale claims
- broad interpretation of the note beyond its declared bounded form

It also excludes any attempt to turn the first matter into a disguised showcase of a private platform, private host, or product surface.

## 8. Relation to the First Execution Slice

This matter directly supports the selected first execution slice.

The current first execution slice is one bounded governed truth-formation attempt for one threshold-eligible matter. This note matter is sufficient for that slice because it can:

- receive real input or contact
- undergo validation
- emit witness trace
- require governance in the selected first cycle
- produce transition and state effects where lawful
- produce registry-preserved canonical artifacts

It is therefore sufficient for the slice to count without broadening the embodiment beyond its intended first boundary.

## 9. Success Relevance

Successful handling of this matter would prove that:

- one real local matter can enter the body
- one bounded claim can pass through the minimum runtime body lawfully
- threshold does not need to collapse into truth by convenience
- governance can remain distinct from validation
- witness can remain distinct from semantic source
- state and transition recording can remain distinct from interface appearance
- canonical preservation can anchor the result of the cycle

This would be enough to prove lawful embodiment in bounded form.

## 10. Failure Relevance

Visible non-completion or failure in this matter would still teach something valid.

- If validation does not pass, the body proves that it can preserve typed non-passage rather than forcing truth.
- If witness is missing, the body proves whether it can keep witness absence visible rather than fabricating trace.
- If governance does not proceed, the body proves whether it can keep lawful force explicit rather than implied.
- If state movement does not occur, the body proves whether non-movement remains visible rather than cosmetically rewritten.
- If canonical preservation fails, the body proves whether preservation failure remains visible rather than being replaced by derivative display.

Failure visibility therefore remains part of the lawful proof rather than a reason to discard the matter as useless.

## 11. Neutrality and Non-Privilege

This matter does not pre-own the architecture for any private entity.

It is chosen for lawful embodiment, not for private capture. It does not imply exclusive early-entry rights, private constitutional ownership, product pre-emption, or privileged implementation status for any vendor, operator, or host.

Its neutrality is one of the reasons it is suitable as the first matter.

## 12. Open Questions

The following remain intentionally open before the first local cycle is actually run:

- the exact input format of the selected note
- the exact operator surface
- the exact host form
- the exact artifact storage path inside the embodiment workspace

These are embodiment-build questions, not constitutional ambiguities.

## 13. Summary

This document defines the first matter of the IAMMAI embodiment layer as one bounded local note matter whose exact received form is the subject of one governed truth-formation attempt in the first lawful local cycle.

Its purpose is to keep the first matter small, neutral, auditable, and sufficient for the minimum runtime body to prove itself without widening into product form, public noise, or private capture.

**One-Line Definition:** `MATTER_DEFINITION.md` defines the first embodiment matter of IAMMAI as one bounded local note matter chosen to exercise the full minimum runtime body in one lawful, auditable cycle.
