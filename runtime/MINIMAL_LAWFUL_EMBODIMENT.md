# IAMMAI Minimal Lawful Embodiment

## 1. Purpose

This file defines what minimal lawful embodiment means for IAMMAI and specifies the first bounded live slice that could prove the body is real without distorting the architecture.

Its job is to define what counts as a first live form, what that first form must include, what it must not pretend to be, and what integrity conditions must still hold if the slice is to count as lawful embodiment rather than cosmetic demonstration.

It does not yet try to define:
- code
- a demo script
- a product plan
- a deployment plan
- infrastructure topology
- broad rollout strategy

It is a bounded runtime and embodiment decision document.

## 2. First Principle

Embodiment must serve the constitutional core.

The first live slice must be smaller than the architecture.  
The first live slice must still be complete enough to count.  
Embodiment must preserve typed distinctions, not perform them cosmetically.

In runtime terms, minimal lawful embodiment means the smallest real live form in which the minimum runtime body can complete one bounded lawful cycle without collapsing validation, witness, governance, state movement, or preservation into one another.

## 3. Why Minimal Lawful Embodiment Is Needed Now

This work should not remain theory only.

- The constitutional core is already published.
- The architecture layer already defines the runtime body and its role separations.
- The implementation layer already defines state, records, witness, governance, validation, transition, and schema structure.
- The runtime layer now defines flow, contract, failure, coordination, and handoff logic in bounded form.

The repo, body, and runtime stack are therefore mature enough for a first live slice.

The first embodiment must still be minimal, bounded, and lawful because early live form should prove the architecture, not distort it. A broad first embodiment would encourage cosmetic continuity, derivative substitution, and premature product framing.

Interpretation should not get ahead of function. The first slice should prove that the body can actually operate, not that it can be narrated attractively.

## 4. What Minimal Lawful Embodiment Means

In repo and runtime terms, minimal lawful embodiment means:

- the smallest real live form that proves the architecture can operate
- not a full system
- not a full product
- not a broad environment
- not a market-facing rollout
- one bounded lawful cycle

It is the minimum live realization of the runtime body in which real contact enters, bounded protocol roles operate distinctly, canonical records are produced or visibly fail to be produced, and the result remains reconstructable.

## 5. Criteria for a Valid First Live Slice

For a first live slice to count as valid minimal lawful embodiment, it must:

- involve real input or contact
- involve validation
- involve witness
- involve governance where the selected transition requires it
- involve state movement where appropriate
- involve canonical preservation
- remain auditable
- remain failure-visible
- remain bounded and non-productized

It should also exercise the minimum runtime body as a body rather than as disconnected parts. A slice that replaces required roles with summary, display, or operator memory does not count as lawful embodiment.

## 6. What the First Live Slice Should Not Be

The first slice should not be:

- a broad platform rollout
- a UI-first demo
- a hype object
- a marketplace
- a product launch
- a dependency-heavy integration
- a private implementation privilege case
- a fake simulation that bypasses real lawful motion

The first slice should be judged by lawful operation, not by surface impressiveness.

## 7. Candidate Shape of the First Live Slice

The recommended first embodiment is a single-matter governed transition cycle.

In bounded form, that means:

1. One bounded matter is selected in advance.
2. One real input or contact enters through the Interface Gateway.
3. The Validator performs the checks required for the selected matter and requested move.
4. The Witness Emitter emits trace for the contact and the checking event where constitutionally relevant.
5. One governance-significant act is requested and resolved in typed form where the selected move requires lawful force.
6. The State Engine performs exactly one lawful movement, or preserves visible non-movement if validation or governance does not permit the move.
7. The Registry / Store preserves the resulting canonical records, including state, transition, validation, witness, and governance records where produced.
8. The slice ends with an auditable canonical record set rather than a derivative display claim.

The exact selected move may be preparation-side or standing-state, but it must be chosen in advance, typed explicitly, and kept bounded to one matter and one cycle. If the chosen move requires governance, the first slice will exercise the full minimum runtime body without widening into a larger system.

## 8. What This First Slice Would Prove

A successful first embodiment would prove:

- the body is not only theory
- the runtime chain can operate lawfully
- typed distinctions survive contact with operation
- witness, validation, governance, state handling, and registry preservation can coordinate without collapse
- failure can remain visible without cosmetic substitution
- canonical records can be produced from real contact rather than narrative summary

It would prove existence of a lawful live body in bounded form, not maturity of a broad system.

## 9. What This First Slice Would Not Yet Prove

This first slice would not yet prove:

- full system completion
- broad deployment readiness
- production scalability
- generalized ambient intelligence
- full ecosystem interoperability
- final infrastructure maturity

It would also not prove that every runtime path, every governance case, or every resolution path has been fully embodied.

## 10. Failure and Integrity Conditions

If the first embodiment is to count as lawful, the following must remain true:

- no silent collapse of roles
- no missing witness disguised as trace
- no validation bypass where validation is required
- no governance-by-vibe
- no state mutation without typed records
- no registry substitution by derivative display
- no cosmetic continuity when the cycle did not actually complete

The slice may end in passage or non-passage. What matters is that the runtime truth of the cycle remains typed, visible, and reconstructable.

## 11. Public / Private Posture

The first embodiment should not grant private privilege.

No private product, vendor, operator, or entity gets constitutional pre-ownership from being the first host of a live slice. The slice may be run in a bounded environment, including an internal one, but its structure must remain neutral, lawful, and non-proprietary in meaning.

The point of first embodiment is not to confer ownership prestige. It is to prove that the body can lawfully exist in motion.

## 12. Open Questions

The following remain intentionally open before a live slice is selected and built:

- the exact slice boundary
- the exact runtime host form
- internal-first versus public-first posture
- how much of the runtime chain, if any, may be stubbed without invalidating the slice
- what success and failure observation mechanism will be used

These are embodiment-design questions, not constitutional ambiguities.

## 13. Summary

This document defines minimal lawful embodiment for IAMMAI as the smallest real live slice in which bounded input, validation, witness, governance where required, state handling, and canonical preservation operate as one lawful cycle.

Its purpose is to set the threshold for first embodiment low enough to be real, but high enough that the architecture is not cosmetically imitated or productized in place of lawful operation.

**One-Line Definition:** `MINIMAL_LAWFUL_EMBODIMENT.md` defines the first bounded live slice of IAMMAI that counts as real embodiment as one auditable lawful runtime cycle that proves the body can operate without collapsing its distinct roles.
