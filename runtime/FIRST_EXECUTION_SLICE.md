# IAMMAI First Execution Slice

## 1. Purpose

This file defines the first bounded live execution slice of IAMMAI.

Its job is to turn the principles in `MINIMAL_LAWFUL_EMBODIMENT.md` into one concrete, minimal, lawful runtime slice that could be embodied without distorting the architecture.

It does not yet try to define:
- code
- a product spec
- a deployment plan
- a broad rollout design
- infrastructure topology
- generalized runtime coverage

It is a bounded runtime and embodiment document that selects and defines one first live slice.

## 2. First Principle

The first execution slice must be real enough to count.

It must stay smaller than the full architecture.  
It must preserve typed distinctions in operation.  
It must remain failure-visible and auditable.

In runtime terms, that means the slice must run as one real attempted lawful cycle rather than as a demo narrative, a display simulation, or a product surface standing in for actual runtime motion.

## 3. Chosen First Slice

The chosen first live slice is:

**One bounded governed truth-formation attempt for one matter already in threshold-eligible condition.**

In practical runtime terms, that means:

- one bounded matter is selected in advance
- that matter is already present in threshold-eligible preparation condition
- one real input or contact enters the body in relation to that matter
- validation checks the selected move
- witness preserves the relevant contact and checking trace
- governance resolves the requested truth-formation attempt in typed form
- the State Engine performs the truth-formation transition only if the upstream basis lawfully permits it
- the Registry / Store preserves the resulting canonical records, or visibly fails to do so

This is one small but complete lawful cycle. It is not a full preparation-chain embodiment, and it is not a broad standing-state system.

## 4. Why This Slice Was Chosen

This slice was chosen because it is minimal enough while still being meaningful.

- It is minimal because it focuses on one matter, one attempted move, one regime boundary, and one bounded runtime cycle.
- It is meaningful because the threshold-to-truth boundary is one of the main anti-collapse boundaries in IAMMAI. Threshold must not silently become truth.
- It is safer than broader first embodiments because it does not require platform surfaces, marketplace assumptions, broad multi-matter handling, or dependency-heavy runtime integration.
- It is capable of proving that the body can operate lawfully because it exercises real input, typed validation, witness trace, governance where required, state movement where permitted, and canonical preservation.

This slice therefore tests the body where convenience pressure would otherwise most easily collapse preparation into standing state.

## 5. Slice Boundaries

### Included

The first slice includes:

- the Interface Gateway for one bounded input or contact
- the Validator for threshold and transition-legality checking, and governance-precondition checking where used
- the Witness Emitter for the relevant contact and checking trace, and later transition-relevant trace where emitted
- the Governor for typed authorization, denial, or HOLD of the truth-formation attempt
- the State Engine for one attempted truth-formation movement and any resulting typed non-movement
- the Registry / Store for canonical preservation of the records actually produced

The slice also includes the canonical artifact families needed for this one cycle:

- validation artifact
- witness artifact
- governance action record
- transition record where movement or containment-relevant effect is produced
- state record where standing state is produced
- registry-preserved canonical records

The outcomes that count inside this slice are:

- authorized truth formation with preserved canonical records
- validation non-passage with preserved non-passage trace
- governance denial with preserved governance trace
- governance HOLD with visible containment effect and preserved trace
- failed or incomplete canonical preservation with visible preservation failure

### Excluded

The first slice excludes:

- broad deployment
- private implementation privilege
- full UI or productization
- large-scale orchestration
- infrastructure complexity
- multi-matter operation
- the full preparation chain from signal through threshold
- broad standing-state behavior beyond this one chosen move
- generalized runtime behavior outside the selected slice

## 6. Runtime Sequence for the Slice

The selected slice runs as the following high-level lawful sequence:

1. One bounded matter is selected in advance in threshold-eligible condition.
2. One real input or contact enters through the Interface Gateway in relation to that matter and the requested truth-formation attempt.
3. The Validator performs the required threshold and transition-legality checks, and any governance-precondition check required for this selected move.
4. The Witness Emitter preserves input-contact trace and validation-related trace where constitutionally relevant.
5. The Governor resolves the requested act in typed form by authorizing, denying, or holding the truth-formation attempt.
6. If the act is authorized and not held, the State Engine performs one lawful truth-formation transition and produces the corresponding transition and state outputs.
7. If the act is denied, held, blocked, or validation does not pass, the non-movement remains typed and visible rather than being rewritten as continuity.
8. The Witness Emitter may also preserve transition-relevant or governance-relevant occurrence where constitutionally relevant and where witness is actually emitted.
9. The Registry / Store preserves the canonical records actually produced by the slice, including visible preservation failure if canonical durability does not complete.

The slice completes when the attempted cycle ends in either:

- preserved lawful movement into truth
- or preserved typed non-passage without cosmetic substitution

## 7. Canonical Artifacts Involved

The canonical artifact families involved in this slice are:

- validation artifact for the bounded checks required by the truth-formation attempt
- witness artifact for contact, validation, and transition-relevant occurrence where witness is emitted
- governance action record for authorize, deny, or HOLD outcome
- transition record for truth-formation movement, and containment-related effect where such effect is recorded
- state record for the resulting truth state where truth formation lawfully occurs
- registry-preserved canonical records for the artifacts actually produced by the slice

No additional artifact family is required for this first execution slice beyond what the selected cycle actually uses.

## 8. Success Conditions

For the slice to count as successful lawful embodiment, the following must be true:

- one typed runtime path was actually attempted
- the required runtime participants operated in their own roles without silent collapse
- the required records for what actually occurred exist in canonical form, or visible non-passage exists where movement did not occur
- validation, witness, governance, state handling, and registry preservation remain distinct
- no silent substitution replaced a missing required role
- no missing canonical preservation is disguised as stored truth
- the result remains auditable from canonical records and visible runtime non-passage

Success in this document does not mean truth formation must always occur. It means the selected slice lawfully ran as a real bounded cycle and remained reconstructable.

## 9. Failure Conditions

The slice must keep the following failure or non-completion surfaces visible:

- validation non-passage
- missing witness where witness was constitutionally relevant
- governance non-progression where governance was required
- failed or blocked state movement
- failed, incomplete, or missing canonical preservation

Failure visibility is part of the proof.

If the slice silently bypasses validation, pretends witness exists when it does not, treats governance non-progression as authorization, mutates state without typed record, or substitutes derivative display for canonical preservation, the slice does not count as lawful embodiment.

## 10. What This Slice Proves

A successful execution of this slice would prove that:

- the IAMMAI body can operate as more than text
- the minimum runtime body can complete one real attempted lawful cycle
- threshold does not need to collapse into truth under operational pressure
- witness, validation, governance, state handling, and registry preservation can coordinate without role confusion
- typed non-passage can remain visible rather than being cosmetically smoothed over
- canonical records can anchor the result of live runtime motion

## 11. What This Slice Does Not Yet Prove

This slice does not yet prove:

- full preparation-chain embodiment
- full standing-state coverage beyond the selected move
- FINALIZE, INVALIDATE, or EVOLVE handling
- continuity, decision, or consequence handling in full runtime breadth
- broad deployment readiness
- production scalability
- full ecosystem interoperability
- final infrastructure maturity

## 12. Public / Private Posture

This first slice does not grant private privilege.

It stands as a neutral architectural embodiment of one lawful runtime cycle. It should not be mistaken for product capture, exclusive early-entry implementation, or constitutional pre-ownership by any host, operator, vendor, or private entity.

The slice proves lawful embodiment only. It does not confer proprietary status over the body.

## 13. Open Questions

The following remain intentionally open before this slice would be built or run:

- the exact host form
- internal-first versus public-first execution
- which concrete runtime surfaces will be used
- what operator or witness visibility is required
- whether any part of the slice is tolerated as simulated, or whether the whole slice must be fully live

These are embodiment-design questions, not constitutional ambiguities.

## 14. Summary

This document defines the first execution slice of IAMMAI as one bounded governed truth-formation attempt for one threshold-eligible matter, carried through real input, validation, witness, governance, state handling, and canonical preservation.

Its purpose is to select one minimal but complete live cycle that can prove the body is real without widening it into product form or collapsing its distinctions for convenience.
